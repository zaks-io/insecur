# ADR-0064: Minimize The Secret-Resident Surface In The Worker Process

Date: 2026-06-03

Status: Accepted

The instance root key and every key the Keyring unwraps under it (Organization Data Keys, Project Data Keys, per-record DEKs, and unlocked Sensitive Values; data keys are wrapped, not derived, per the ADR-0005 Amendment (2026-06-03)) are reachable in the Worker process only for the span of the single request that needs them, never as module-global state and never as a `process.env` value in production. This narrows the window in which in-process key material is reachable from the lifetime of the V8 isolate, which serves many requests over minutes, to the lifetime of one request handler. ADR-0031 already requires that the Keyring's cache of unlocked keys is scoped per tenant so nothing leaks across tenants in the shared isolate; this ADR adds the orthogonal time-and-placement bound, and reconciles the code, which today caches a process-global Keyring in `packages/crypto`'s `crypto-runtime.ts` and falls back to a plaintext `INSECUR_INSTANCE_ROOT_KEY_HEX` environment variable.

The motivating threat is in-process code execution: a compromised dependency or a gadget that can read process-resident state. A key pinned in a module-global, or a root key sitting in `process.env`, is reachable for the whole isolate lifetime, so one read compromises all future decrypts. Constructing the Keyring inside the request handler and letting it fall out of scope when the request ends, and sourcing the root key from the Cloudflare Secrets Store binding through an on-demand `get()` rather than from the environment, mean the material is resident only across the fetch-unwrap-decrypt span of an active operation. The Keyring object itself may be constructed per request; what is forbidden is module-global retention of key material or anything unwrapped under it.

In production the plaintext-environment root-key fallback is refused, not merely deprioritized. `INSECUR_INSTANCE_ROOT_KEY_HEX` is a development-only convenience; when the Worker runs in production it fails closed at first use with `RootKeyNotConfiguredError` rather than reading a root key from the environment, because an environment-resident root key defeats the entire surface-minimization this ADR exists to provide. The Secrets Store binding (ADR-0028) is the only accepted production root-key source.

This is a window-narrowing decision, not a claim of in-process secrecy, and the limit is stated so nothing downstream oversells it. During the request that is actively decrypting, the keys and the resulting plaintext are necessarily in memory; in-process code execution landing inside that window can still reach them. Closing that residual window is not the Keyring's job and is not attempted here: it belongs to not running untrusted code in the isolate, to the Storage Security Gate failing closed, and to Secret-Free Logging and audit. The wall this ADR builds is against long-lived resident material and against an environment-readable root key, not against arbitrary code already executing mid-decrypt.

V1 keeps a single instance root key under which the whole chain is wrapped, with no per-tenant or customer-supplied root material. A Self-Hosted Instance generates, loads, and escrows its own instance root key (ADR-0028), which is a different root per instance, not a different root per tenant within an instance. The named future path to per-tenant or customer-controlled root material is Customer-Managed Key Custody on a Hosted Instance (ADR-0050), whose trigger and migration are recorded there; this ADR's per-request resolution model is the seam that makes that addition non-breaking, because a per-tenant or customer-granted wrapping authority slots into the same on-demand, request-scoped resolution the single root already uses. Until that trigger fires, V1 is single instance root, no Bring Your Own Key.

## Consequences

Per-request Keyring construction and an on-demand root fetch replace the module-global cache in `crypto-runtime.ts`; the change is mechanical but load-bearing, and "no module-global key material" and "production refuses the environment root-key fallback" are each a unit test against fakes for the Secrets Store binding and the environment. The development plaintext-environment path stays, gated to non-production, so local iteration is unaffected.

The residual in-request exposure window is an accepted, documented non-goal, not an oversight. Stating it keeps the Storage Security Gate, Secret-Free Logging, and the no-untrusted-code posture as the owners of that window rather than letting Keyring scoping be mistaken for a complete defense.

Because the single-instance-root model and the per-request resolution seam are chosen together, adding Customer-Managed Key Custody (ADR-0050) later rewraps both Organization Data Keys and Project Data Keys onto a customer-granted authority without re-architecting how the runtime resolves a key, since resolution is already request-scoped and material is already never module-global.

## Amendment (2026-06-11): Customer-Managed Key Custody trigger lives in the phasing parking lot

The record above says Customer-Managed Key Custody's "trigger and migration are recorded" in ADR-0050. ADR-0050 records the migration mechanics only; it names no promotion trigger. The promotion trigger is recorded in the [phasing.md](../phasing.md) deferred scope parking lot row for Customer-Managed Key Custody (ADR-0050), which is the source of truth for deferred work and its promotion conditions. The migration mechanics stay in ADR-0050.
