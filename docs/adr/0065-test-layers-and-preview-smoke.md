# ADR-0065: Test Layers And Preview-Env Smoke

Date: 2026-06-03

Status: Accepted

insecur's automated tests are organized into layers, each defined by where its Postgres comes from and what failure class it is meant to catch: unit tests with no database, integration and RLS tests against a Docker Compose Postgres that runs locally and in CI, and a preview-environment smoke that runs the First Value loop over HTTP against a deployed Worker backed by a shared Neon preview branch through a Cloudflare Hyperdrive binding. The layers are deliberately distinct because they answer different questions: the unit layer asks whether the logic is correct, the integration and RLS layer asks whether the composed code round-trips through real Postgres with row-level security forced, and the preview smoke asks whether a real deploy actually serves with real bindings and real secrets. A green run in one layer does not stand in for another, and the preview smoke in particular is the only layer that can catch a broken deploy, a missing binding, or a bad secret, because those failures do not exist until the code runs in the deployed runtime.

The composed First Value loop now has a real end-to-end test at the integration layer: `apps/worker/test/e2e/first-value-loop.e2e.test.ts` drives the actual Hono route stack with `app.request` against Docker Compose Postgres and real crypto, with no package mocks, and asserts that a secret value written through `POST .../secrets/by-variable-key` round-trips back out through grant issue and grant consume. This is the seam the per-package integration suites cannot cover, because each of those exercises one package while the existing `fv12-routes.test.ts` mocks the package layer to test routing in isolation. A remote agent can run this loop with `pnpm dev:db:reset && pnpm test:e2e` using only Docker, with no Cloudflare or Neon credentials, which is the property that makes it useful for agents iterating in their own sandboxes.

Miniflare and the Cloudflare Workers Vitest pool are deliberately not used for this layer. The persistence seam in `@insecur/tenant-store` connects with the `postgres` (Postgres.js) driver over a raw TCP socket, which workerd does not provide locally without a Hyperdrive binding, so a "real" First Value loop cannot reach the database from inside the workers pool against a local Postgres. Running the loop in plain Node Vitest against Docker Compose Postgres exercises every line of real product code that matters and is available today; a workerd-runtime test would either mock persistence (proving routing, not the round trip) or require the cloud, so it buys a faithful runtime at the cost of the thing the test exists to prove. Miniflare is therefore deferred, not rejected; if a workerd-specific regression class emerges, a thin runtime-compatibility check can be added without disturbing the loop test.

The Worker reaches Postgres through a Cloudflare Hyperdrive binding, keeping the `postgres` driver unchanged. The alternative, the `@neondatabase/serverless` HTTP driver, was rejected for V1 because it would force `tenant-store` to carry two driver paths (Postgres.js locally, serverless HTTP in the Worker). Keeping a single driver and pooling it through Hyperdrive means the same `tenant-store` code runs in tests, in preview, and in production, and the binding is the only thing that changes between environments.

The preview smoke layer targets a shared preview environment, not pull requests. PRs must not create Neon branches, Hyperdrive configs, or per-PR Worker deploys because the repository cannot support unbounded Neon preview environments. The local and hosted PR gate is the Docker Compose `postgres-integration` job. `scripts/ci/smoke-first-value.mjs` remains the shared preview smoke entrypoint and hard-fails when `SMOKE_BASE_URL` is unset, because a smoke that silently passes against nothing is worse than no smoke. `pnpm deploy:preview` and the `Deploy Preview` workflow deploy the shared preview Workers; the shared preview smoke runs as an explicit post-deploy check after the live Neon preview branch and smoke admission are configured.

The integration and RLS layers are wired into CI through the existing `postgres-integration` job, which provisions Docker Compose Postgres 17 on a Blacksmith runner, runs `assert:rls-credentials` (distinct migration/runtime URLs, runtime `NOBYPASSRLS`), then runs every workspace DB-backed `test:rls` suite with `INSECUR_CI_RLS_GATE=1` so skipped suites fail the build. Turbo `test:rls` fans out to `@insecur/tenant-store` (forced-RLS plus tenant-store root `*.integration.test.ts` suites) and the package-level DB-backed `*.integration.test.ts` suites in `@insecur/access`, `@insecur/audit`, `@insecur/operations`, `@insecur/onboarding`, `@insecur/secret-store`, `@insecur/runtime-injection`, `@insecur/machine-auth`, and `@insecur/instance-bootstrap`. `test:e2e` runs the First Value loop and `test:canary` runs the no-plaintext canary gate in the same DB-provisioned CI path. Ordinary package `test` tasks exclude DB-backed integration suites so `pnpm verify` remains unit-only even when `DATABASE_URL_RUNTIME` exists locally.

## Consequences

The First Value loop now has a single test that fails if any part of the composed write-encrypt-persist-grant-consume-decrypt path regresses, runnable by agents and in CI with only Docker, which closes the gap where each package was tested in isolation but the composition was not. RLS tests now execute in CI rather than skipping, narrowing the window where a tenant-isolation regression could merge unnoticed.

The preview smoke layer's value is exercised on explicit shared-preview deploys, not on every PR. The cost is that deployed-runtime regressions are not caught on every PR; that is accepted because unbounded Neon branches and Worker deploys per PR are not operationally sustainable for this repository.

Choosing Hyperdrive with the unchanged `postgres` driver keeps `tenant-store` single-driver across all three layers, so the preview and production binding work also de-risks production: the connection topology that the preview smoke exercises is the same one production will use, and there is no second driver path to test or maintain.
