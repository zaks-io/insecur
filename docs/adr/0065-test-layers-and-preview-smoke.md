# ADR-0065: Test Layers And Preview-Env Smoke

Date: 2026-06-03

Status: Accepted

insecur's automated tests are organized into three layers, each defined by where its Postgres comes from and what failure class it is meant to catch: unit tests with no database, integration and RLS tests against a Docker Compose Postgres that runs locally and in CI, and a preview-environment smoke that runs the First Value loop over HTTP against a deployed Worker backed by an ephemeral Neon branch through a Cloudflare Hyperdrive binding. The layers are deliberately distinct because they answer different questions: the unit layer asks whether the logic is correct, the integration and RLS layer asks whether the composed code round-trips through real Postgres with row-level security forced, and the preview smoke asks whether a real deploy actually serves with real bindings and real secrets. A green run in one layer does not stand in for another, and the preview smoke in particular is the only layer that can catch a broken deploy, a missing binding, or a bad secret, because those failures do not exist until the code runs in the deployed runtime.

The composed First Value loop now has a real end-to-end test at the integration layer: `apps/worker/test/e2e/first-value-loop.e2e.test.ts` drives the actual Hono route stack with `app.request` against Docker Compose Postgres and real crypto, with no package mocks, and asserts that a secret value written through `POST .../secrets/by-variable-key` round-trips back out through grant issue and grant consume. This is the seam the per-package integration suites cannot cover, because each of those exercises one package while the existing `fv12-routes.test.ts` mocks the package layer to test routing in isolation. A remote agent can run this loop with `pnpm dev:db:reset && pnpm test:e2e` using only Docker, with no Cloudflare or Neon credentials, which is the property that makes it useful for agents iterating in their own sandboxes.

Miniflare and the Cloudflare Workers Vitest pool are deliberately not used for this layer. The persistence seam in `@insecur/tenant-store` connects with the `postgres` (Postgres.js) driver over a raw TCP socket, which workerd does not provide locally without a Hyperdrive binding, so a "real" First Value loop cannot reach the database from inside the workers pool against a local Postgres. Running the loop in plain Node Vitest against Docker Compose Postgres exercises every line of real product code that matters and is available today; a workerd-runtime test would either mock persistence (proving routing, not the round trip) or require the cloud, so it buys a faithful runtime at the cost of the thing the test exists to prove. Miniflare is therefore deferred, not rejected; if a workerd-specific regression class emerges, a thin runtime-compatibility check can be added without disturbing the loop test.

The Worker reaches Postgres through a Cloudflare Hyperdrive binding, keeping the `postgres` driver unchanged. This matches the proven preview-environment pattern in the sibling `agent-paste` repository, where a per-PR Neon branch is bound through a per-PR Hyperdrive to a per-PR Worker deploy. The alternative, the `@neondatabase/serverless` HTTP driver, was rejected for V1 because it would force `tenant-store` to carry two driver paths (Postgres.js locally, serverless HTTP in the Worker) and diverge from the topology already validated in agent-paste. Keeping a single driver and pooling it through Hyperdrive means the same `tenant-store` code runs in tests, in preview, and in production, and the binding is the only thing that changes between environments.

The preview smoke layer is built but gated off, because the infrastructure it targets does not exist yet. There is no deployed `-dev` or preview Worker, no `env.preview` Hyperdrive binding wired into the Worker's connection resolution, and no Neon or Cloudflare credentials configured in CI. The `pr-preview.yml` and `pr-preview-cleanup.yml` workflows, the `scripts/ci/create-hyperdrive.mjs`, `delete-neon-pr-branch.mjs`, and `smoke-first-value.mjs` scripts, and the `env.preview` scaffold in `apps/worker/wrangler.jsonc` are all present and lint-clean, but the workflows run only when the repository variable `PREVIEW_ENV_ENABLED` is `'true'`, so they stay inert on normal PRs. Within the smoke job the `smoke-first-value.mjs` script hard-fails when `SMOKE_BASE_URL` is unset rather than skipping, because a smoke that silently passes against nothing is worse than no smoke. The gate is therefore at the job level (opt-in until infra lands), and the hard-fail is at the script level (no green-by-skip once the job runs). Standing up the preview environment, wiring the Worker to read its connection from the `env.DB` Hyperdrive binding instead of resolving ambiently, and flipping `PREVIEW_ENV_ENABLED` is tracked as INS-164 (with the Hyperdrive binding shared with INS-162) and is the prerequisite that turns this layer live.

The integration and RLS layers are wired into CI through the existing `postgres-integration` job, which already provisions Docker Compose Postgres on a Blacksmith runner. The job now runs `test:rls` (the forced-RLS tenant suite, which previously was green-by-skip in CI per the INS-144 backlog) and `test:e2e` (the First Value loop) in addition to the instance-bootstrap integration suite. The broad raw-SQL package integration run is intentionally not wired in yet: those queries are being migrated to Drizzle and are in flux, and wiring them surfaced a pre-existing fixture bug in `packages/access/test/resolve-machine-effective-access.integration.test.ts` (a `sql.array(..., "text")` call that produces a malformed array literal) that had been hidden by green-by-skip. Fixing that bug and wiring the broad integration run belong with the Drizzle conversion, not ahead of it, and are recorded on the same follow-up ticket.

## Consequences

The First Value loop now has a single test that fails if any part of the composed write-encrypt-persist-grant-consume-decrypt path regresses, runnable by agents and in CI with only Docker, which closes the gap where each package was tested in isolation but the composition was not. RLS tests now execute in CI rather than skipping, narrowing the window where a tenant-isolation regression could merge unnoticed.

The preview smoke layer's value is latent until the preview environment exists; shipping it gated means the conventions, scripts, and workflow shape are designed in from the start rather than bolted on later, and the work to make it live is reduced to standing up infrastructure and flipping a flag rather than authoring the harness under deadline. The cost is carrying workflow and script files that do nothing until then, which is accepted as the price of not faking a passing smoke against absent infrastructure.

Choosing Hyperdrive with the unchanged `postgres` driver keeps `tenant-store` single-driver across all three layers, so the preview and production binding work also de-risks production: the connection topology that the preview smoke exercises is the same one production will use, and there is no second driver path to test or maintain.
