# ADR-0038: Protected Environment Delivery Requires A Machine Credential

Date: 2026-05-24

Status: Accepted

Issuing an Injection Grant, or performing any Secret Delivery, for a Protected Environment requires a Machine Identity credential bound to that environment, a deploy key or an OIDC identity. A human session token, including the short-lived `INSECUR_SESSION_TOKEN` a coding agent inherits when it runs inside a human session (ADR-0032), cannot obtain a Protected Environment Injection Grant. Runtime Injection is a read path: the child process the caller controls can read its delivered environment once it crosses the Runtime Trust Boundary, so the ability to inject is the ability to read. Possession of a credential the agent does not hold, rather than agent restraint or a per-run challenge, is what prevents a local agent from reading Protected Environment Sensitive Values. The deploy credential lives in CI/CD where the agent is not, so the agent holds the human's session token but never the credential that unlocks protected delivery.

Protection is a property an environment carries, not a name (Protected Environment is an environment whose secrets do not support Secret Reveal). Development is non-protected so the local agent iteration loop stays permissive. Staging and production are Protected. Preview defaults to Protected and may be opted down to the non-protected development tier per project only when its values are confirmed non-production. The risk is asymmetric: misclassifying a real-value environment as non-protected leaks live secrets to an agent, while misclassifying a throwaway environment as protected only adds friction, so the default favors protection.

Creating, rotating, or attaching a Protected Environment deploy credential stays a High-Assurance Challenge action and is never autonomous, otherwise an agent holding the session token would simply mint its own credential and defeat the boundary. OIDC is preferred wherever the deploy platform supports it because it removes any static key to store, read, or mint; a deploy key is the fallback when OIDC is unavailable, consistent with ADR-0004.

## Considered Options

We considered gating Protected Environment injection from a human session behind a fresh High-Assurance Challenge, rejected because the agent controls the child process: a human clearing the challenge authorizes the human's presence but the plaintext still lands in an agent-controlled process and is harvested, so step-up proves presence, not custody. We considered leaving protected injection reachable by the inherited human session token, the status quo, rejected because it hands a local agent a full read path to production through inject-and-print, a back-door Secret Reveal on an environment that by definition does not support Secret Reveal. We considered barring protected injection for every caller including machine credentials, rejected because it walls off CI/CD, the legitimate consumer, which is non-interactive and cannot step up. We considered classifying protection by environment name, rejected because protection is a property and a preview environment can be either real-value or throwaway, so hardcoding names misclassifies realistic previews.

## Consequences

A coding agent operating in a human session can do everything in non-protected environments but has no path to Protected Environment Sensitive Values, enforced by possession rather than restraint, which extends ADR-0032 and the delivery-first posture of ADR-0016 and ADR-0019. Protected delivery becomes a CI/CD concern, and local human runs against production-grade values are intentionally unavailable; realistic local work uses a non-protected environment with development-grade values. Deploy-credential custody is now load-bearing, so minting and rotation stay High-Assurance Challenge actions and OIDC is preferred to remove static keys. Production protected delivery still also requires the Storage Security Gate; the machine-credential requirement is an additional, orthogonal gate, not a replacement.

## Amendment (2026-06-11): Machine-credential requirement scoped to Runtime Injection

The opening sentence above says issuing an Injection Grant "or performing any Secret Delivery" for a Protected Environment requires a Machine Identity credential. The requirement is scoped to Runtime Injection Grants, the delivery shape this ADR reasons about: plaintext lands in a caller-controlled child process, so the ability to inject is the ability to read, and credential possession is the boundary. Server-side Secret Sync to providers is a different shape — plaintext flows from insecur to the provider's secret store, never into a caller-controlled process — and is instead governed by a High-Assurance Challenge plus the protected delivery configuration approval path in the Human Approval Surface; protected Secret Sync enable and run are excluded from agent-reachable channels (ADR-0042, docs/cli-and-sync.md). This matches product-spec section 7, which states the machine-credential rule for Protected Environment Runtime Injection only. A human can therefore trigger a protected Secret Sync run through the approval path without holding a deploy credential; what a human session still cannot obtain is a Protected Environment Injection Grant.

## Amendment (2026-06-12): Protected issuance is a machine-only Authorization Scope, not an actor-type check

The denial above is enforced by a distinct Authorization Scope atom, `runtime_injection:grant_issue_protected`, registered in `packages/access/src/authorization-scopes.ts`, not by inspecting the caller's actor type. The existing `runtime_injection:grant_issue` atom sits in human Built-In Role bundles (owner, admin, and developer in `packages/access/src/built-in-role-scopes.ts`), so a single caller-agnostic check on that atom cannot express this ADR's denial, and checking `actor.type` instead would be exactly the regression ADR-0034 exists to prevent: routes and services never branch on actor type or Role name. The new atom dissolves the conflict by construction, because the grant service checks exactly one scope against the resolved Effective Access set and the set itself is what differs between a human session and a machine credential.

Two structural properties make the atom machine-only. First, human effective-access resolution can never produce it. The atom appears in no Built-In Role bundle, and human scopes today derive solely from `rolePreset` expansion (`expandBuiltInRolePresetToScopes` over membership rows), so absence from every bundle is absence from every human Effective Access set; the invariant is the resolution-path property, not the bundle snapshot, and any future custom-role or membership-scope feature must re-prove that the human path cannot contribute the atom before it ships. Second, the atom is contributed only by machine effective-access resolution, the Machine Identity Memberships intersect token scope intersect Credential Scopes path AG7 owns (`packages/access/src/build-machine-effective-access.ts`), and only when the credential is bound to the target Environment. A registry conformance assertion pins both invariants, that the atom is absent from every human Built-In Role bundle and absent from `MACHINE_FORBIDDEN_AUTHORIZATION_SCOPES` in `packages/access/src/machine-forbidden-scopes.ts`, so weakening the boundary in either direction requires a red diff to the registry and its test.

The Effective Access Resolver stays protected-ignorant per ADR-0034. The Runtime Injection Grant Service, not the resolver, maps the target Environment's protection property to the one scope the action requires: non-protected issuance requires `runtime_injection:grant_issue`, protected issuance requires `runtime_injection:grant_issue_protected` only. That mapping branches on a resource property, the protection classification this ADR already defines as a property the environment carries, never on actor type, satisfying both this ADR and the no-actor-branch rule by construction. Machine resolution contributes the atom purely from credential-to-environment binding and never reads protection classification, so no protected-awareness leaks below the grant service. The grant-issue service in `packages/runtime-injection/src/issue-injection-grant.ts` currently has no protected handling and assumes a user actor; this amendment is the mechanism it implements.

The boundary applies at grant issuance, matching the opening paragraph's "issuing" framing. Consume inherits it through the grant itself: an Injection Grant is one-use and bound at issue time, so a grant that could not be issued cannot exist to be consumed. Deploy-key bounds to explicit Runtime Policy Key IDs (ADR-0004) remain an orthogonal AG7 credential constraint on what a deploy key may reach and are not folded into this scope atom.
