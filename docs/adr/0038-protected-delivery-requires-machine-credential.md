# ADR-0038: Protected Environment Delivery Requires A Machine Credential

Date: 2026-05-24

Status: Accepted

Issuing an Injection Grant, or performing any Secret Delivery, for a Protected Environment requires a Machine Identity credential bound to that environment, a deploy key or an OIDC identity. A human session token, including the short-lived `INSECUR_SESSION_TOKEN` a coding agent inherits when it runs inside a human session (ADR-0032), cannot obtain a Protected Environment Injection Grant. Runtime Injection is a read path: the child process the caller controls can read its delivered environment once it crosses the Runtime Trust Boundary, so the ability to inject is the ability to read. Possession of a credential the agent does not hold, rather than agent restraint or a per-run challenge, is what prevents a local agent from reading Protected Environment Sensitive Values. The deploy credential lives in CI/CD where the agent is not, so the agent holds the human's session token but never the credential that unlocks protected delivery.

Protection is a property an environment carries, not a name (Protected Environment is an environment whose secrets do not support Secret Reveal). Development is non-protected so the local agent iteration loop stays permissive. Staging and production are Protected. Preview defaults to Protected and may be opted down to the non-protected development tier per project only when its values are confirmed non-production. The risk is asymmetric: misclassifying a real-value environment as non-protected leaks live secrets to an agent, while misclassifying a throwaway environment as protected only adds friction, so the default favors protection.

Creating, rotating, or attaching a Protected Environment deploy credential stays a High-Assurance Challenge action and is never autonomous, otherwise an agent holding the session token would simply mint its own credential and defeat the boundary. OIDC is preferred wherever the deploy platform supports it because it removes any static key to store, read, or mint; a deploy key is the fallback when OIDC is unavailable, consistent with ADR-0004.

## Considered Options

We considered gating Protected Environment injection from a human session behind a fresh High-Assurance Challenge, rejected because the agent controls the child process: a human clearing the challenge authorizes the human's presence but the plaintext still lands in an agent-controlled process and is harvested, so step-up proves presence, not custody. We considered leaving protected injection reachable by the inherited human session token, the status quo, rejected because it hands a local agent a full read path to production through inject-and-print, a back-door Secret Reveal on an environment that by definition does not support Secret Reveal. We considered barring protected injection for every caller including machine credentials, rejected because it walls off CI/CD, the legitimate consumer, which is non-interactive and cannot step up. We considered classifying protection by environment name, rejected because protection is a property and a preview environment can be either real-value or throwaway, so hardcoding names misclassifies realistic previews.

## Consequences

A coding agent operating in a human session can do everything in non-protected environments but has no path to Protected Environment Sensitive Values, enforced by possession rather than restraint, which extends ADR-0032 and the delivery-first posture of ADR-0016 and ADR-0019. Protected delivery becomes a CI/CD concern, and local human runs against production-grade values are intentionally unavailable; realistic local work uses a non-protected environment with development-grade values. Deploy-credential custody is now load-bearing, so minting and rotation stay High-Assurance Challenge actions and OIDC is preferred to remove static keys. Production protected delivery still also requires the Storage Security Gate; the machine-credential requirement is an additional, orthogonal gate, not a replacement.
